# WP-0.2 decisions

Interpretations made to unblock implementation. Not ADR material; revisit if wrong.

1. **"One interface" (AC).** A thin wrapper over `database/sql` (`storage.DB` = `*sql.DB` + `Dialect` + a `Rebind` helper for the `?` â†’ `$1` placeholder gap), not a bespoke repository/ORM interface. Matches ADR-002: "a thin storage adapter layer covers the (small) SQL dialect gap."
2. **Postgres driver.** `jackc/pgx/v5` in stdlib (`database/sql`) mode â€” pure Go, no CGO, consistent with CLAUDE.md's no-CGO rule already applied to `modernc.org/sqlite`. Not treated as a "new runtime dependency" needing an ADR: ADR-002/ADR-015 already mandate Postgres support, a driver is required to fulfill it.
3. **Migration runner.** Hand-rolled, not a third-party library (golang-migrate etc.) â€” per ponytail's stdlib-first ladder and CLAUDE.md's no-new-heavyweight-deps rule. Embedded `.sql` files applied forward-only in lexical filename order, tracked in a `schema_migrations` bookkeeping table (kernel bookkeeping, not tenant data â€” no `tenant_id`). Expand/contract discipline (docs/03 Â§Conventions) is enforced by convention + PR review at this WP; automated destructive-DDL detection is a candidate for WP-0.8's gauntlet, not built here.
4. **Testcontainers harness.** `github.com/testcontainers/testcontainers-go` (postgres module) for ephemeral Postgres 18 in tests/CI â€” explicitly named in the WP-0.2 roadmap line, no ADR needed. SQLite adapter tests use a tmp-file DB; no container required since it's embedded. Version per ADR-002 amendment 2026-07-07 (16+ â†’ 18+, Dan).
5. **"Identical adapter conformance suite" (AC).** One shared Go test-helper package (`kernel/storage/conformance`) exposing `Run(t *testing.T, db *storage.DB)`; both the Postgres and SQLite adapter packages call the exact same function from their own `_test.go` files. "Identical" is satisfied by construction, not by parallel-but-separate test files.
6. **Invariants (docs/19).** No INV-* is implemented or registered at this WP. No domain or tenant tables exist yet â€” tenancy is WP-0.3, the event store is WP-0.4. This WP builds the substrate (adapter + migration runner) that the Integrity Gauntlet's "Adapter conformance" and "Migration integrity" suites (docs/19 Â§3) will run against starting with WP-0.8/WP-0.4.
7. **`tenant_id` on the conformance fixture table.** The fixture table created by this WP's own migration includes a `tenant_id` column (first index column, per docs/03 convention) to prove the adapter/migration path carries it end-to-end â€” but RLS policy wiring is deferred to WP-0.3, since no tenant-context middleware exists yet to populate the session context RLS would check against.
8. **CI.** Postgres conformance tests require Docker; available by default on GitHub-hosted `ubuntu-latest` runners, so they run in the existing `test` job with no new CI service block. Not a solo-mode/shipped-binary requirement â€” only dev/CI machines run these tests.
9. **`schema_migrations.applied_at` is UTC by construction, not by column type alone.** Uses `TIMESTAMPTZ` (not `TIMESTAMP`) and binds an explicit `time.Now().UTC()` Go value rather than SQL `CURRENT_TIMESTAMP`, per integrity review: a bare `CURRENT_TIMESTAMP` cast into a tz-less `TIMESTAMP` column is session-timezone-dependent on Postgres, which would silently violate CLAUDE.md's "Time: UTC in storage, always" rule on a self-hosted deployment with a non-UTC session timezone.
10. **No advisory lock around `Apply`'s check-then-insert.** Two processes starting concurrently could both observe a migration as unapplied and race to run it; on Postgres the loser hits the `schema_migrations` PK conflict and its transaction rolls back cleanly (DDL is transactional), so this is not corruption-prone. Flagged by integrity review as a backlog item â€” not required for this WP's AC, revisit before the runner is relied on by multi-instance deployments (docs/22).
